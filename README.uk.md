# Project & Task Manager

SPA для управління проектами та завданнями всередині них. Vue 3 (Composition API) + TypeScript (strict, без `any`) + Pinia. Дані працюють без реального бекенду — через мок-адаптер Axios поверх `localStorage`.

---

## Можливості

- **Список проектів:** таблиця з сортуванням за будь-якою колонкою, пошуком за назвою, фільтром за статусом, зміною ширини колонок. Створення / редагування / видалення проекту без перезавантаження.
- **Сторінка проекту, два режими:** Таблиця і Канбан над **спільними** даними стору — зміни синхронізуються в реальному часі в обидва боки. Обраний режим зберігається між перезавантаженнями.
- **Таблиця завдань:** сортування за терміном і статусом, фільтрація за виконавцем і статусом, drag-and-drop рядків, зміна ширини колонок.
- **Канбан:** три колонки-статуси; перетягування всередині колонки змінює порядок, між колонками — статус і позицію.
- **Форми** з валідацією (zod + vee-validate), помилки on blur і on submit.
- **Бонуси:** toast-сповіщення, donut-діаграма розподілу завдань за статусами, збереження фільтрів/сортування в URL (shareable & bookmarkable), світла/темна тема, анімації.

## Стек

Vue 3 (Composition API, `<script setup>`) · TypeScript (strict) · Pinia · Vue Router · Axios · SCSS · vue-draggable-plus · zod + vee-validate · Vite.

## Локальний запуск

```bash
npm install
npm run dev         # dev-сервер (http://localhost:5173)
npm run build       # продакшн-збірка
npm run preview     # перегляд продакшн-збірки локально
npm run lint        # ESLint (фейлиться на будь-якому `any`)
npm run type-check  # vue-tsc --noEmit
```

Бекенд не потрібен — застосунок повністю самодостатній (мок-адаптер + `localStorage`). При першому запуску підіймаються сід-дані; після цього стан зберігається в браузері.

## Архітектура

```
src/
  api/          # Axios-шар: generic-типізовані обгортки, мок-адаптер, обробка помилок
  stores/       # Pinia-модулі (projects, tasks) — CRUD-екшени, похідний стан
  composables/  # перевикористовувана реактивна логіка (sort, resize, drag, filters, …)
  utils/        # чисті функції (формат дати, плюралізація, лейбли статусів, aria)
  types/        # доменні моделі, enum-и (const-object), DTO
  components/   # компоненти за відповідальністю + ui/ примітиви + icons/
  views/        # сторінки (ProjectsView, ProjectView)
  router/       # маршрути vue-router
  styles/       # SCSS: токени, reset, mixins, змінні
  App.vue       # кореневий компонент
  main.ts       # точка входу: Pinia, router, глобальні стилі
```

### API-шар (`src/api/`)

Усі дані йдуть **виключно** через Axios — навіть без реального бекенду. `http` надає generic-типізовані обгортки (`get<T>`, `post<T, B>`, …), що повертають готовий payload, тож компоненти не торкаються `AxiosResponse`.

Обробка помилок зосереджена **в одному місці**: response-interceptor нормалізує будь-який збій у `ApiError`, а `settle()` загортає виклик у tagged-результат `{ ok, data } | { ok, error }`. Це **єдиний** `try/catch` навколо запитів — стори і компоненти лише гілкуються по `result.ok` і ніколи не обгортають axios власним `try/catch`.

Реалізовані всі 8 ендпоінтів: `GET/POST/PUT/DELETE /projects`, `GET /tasks` (з опційним `?projectId=` — без параметра повертає всі завдання, потрібно для колонки лічильника на сторінці проектів), `POST/PUT/DELETE /tasks`.

### Стор (`src/stores/`)

Окремі модулі для проектів і завдань. Завдання — єдине джерело істини для обох режимів відображення: мутації застосовуються до стану одразу (оптимістично), тож зміна статусу/порядку відображається скрізь миттєво, а потім персиститься через API. Лічильники завдань і розподіл за статусами — **похідний реактивний** стан (`computed`), не зберігається на моделі.

### Composables

Уся логіка таблиць/drag/фільтрів винесена з компонентів у хуки:

- `useTableSort` — дженерик-сортування, три-станний цикл по колонці (inactive → asc → desc → manual).
- `useColumnResize` — зміна ширини колонок (керує grid-треками).
- `useDragSort` / `useKanbanDrag` — обв'язка drag-and-drop для списку та для board. Тримають лише writable-дзеркало для `<VueDraggable>` і репортять новий порядок; **не** виводять `order`/`status` самі — це робить стор.
- `useFilters` — дженерик-фільтрація (текстовий пошук + exact-match фільтри, у т.ч. tri-state).
- `useNullableSelect` — міст між nullable-фільтром у сторі та non-null моделлю `<select>`.
- `useQueryFilters` — двостороння синхронізація фільтрів/сортування з query-рядком URL.
- `usePersistedRef`, `useTheme`, `useToast` — persistence, тема, сповіщення.

Чисті (нереактивні) операції винесені окремо в `utils/` — формат дати, українська плюралізація, лейбли статусів (єдине джерело правди), `aria-sort`.

### Типізація

Доменні моделі та DTO (через `Omit`/`Partial`) в одному місці. Статуси — **const-object + derived union**, а не TS-enum: значення лишаються звичайними рядками (прямо біндяться в `<option>` і round-trip-ляться через API/JSON без числового reverse-mapping), об'єкт ітерується для рендера опцій, і він повністю erasable під `verbatimModuleSyntax`. Generic-типізація наскрізна в API-шарі та composables. `any` заборонено правилом ESLint (`no-explicit-any: error`) — у коді 0 входжень.

## Мок-бекенд

Реального сервера немає. Кастомний **Axios-адаптер** (`src/api/mockAdapter.ts`) перехоплює кожен запит і обслуговує його з `localStorage`, додаючи випадкову затримку **150–300 мс** для імітації мережі. Кастомний адаптер обрано замість `axios-mock-adapter`, щоб мок був самодостатнім і повністю типізованим (без зайвої залежності й без `any`).

Маршрути матчаться за методом і URL, тіло парситься з JSON, відповіді типізовані. Видалення проекту каскадно прибирає його завдання. Для коду застосунку це прозоро — він робить звичайні типізовані Axios-виклики й нічого не знає про мок.

## Свідомі рішення та компроміси

- **Оптимістичні мутації без rollback.** `reorderTasks`/`syncBoard` міняють стан до відповіді сервера (миттєвий UI), а при невдалому `PUT` показують помилку без відкату. Мок не фейлить, тож для цього завдання rollback зайвий; у проді тут був би відкат стану при помилці запиту.
- **Reorder шле кілька `PUT`.** Зміна порядку персиститься по одному `PUT /tasks/:id` на кожне змінене завдання (`Promise.all`). На реальному бекенді доречнішим був би batch-ендпоінт (напр. `PATCH /tasks/reorder`); у межах мока це свідомо спрощено.
- **Фільтри/сортування — в URL, режим відображення — в `localStorage`.** Стан *перегляду* (фільтри, сорт) живе в query-рядку: він shareable, bookmarkable і дружить з back/forward. Вибраний режим Таблиця/Канбан тримається в `localStorage` — як прямо вимагає завдання. Один механізм на кожен тип стану, не змішано.
- **Два окремі drag-композабли** (`useDragSort` для списку, `useKanbanDrag` для board) замість одного дженерика. Спільне ядро — кілька рядків; об'єднання в `useDragSort<T | Record<S, T[]>>` зробило б обидва сценарії менш читабельними. Семантика різна (плоский список vs `Record<status, list>`), тож вони лишені роздільними навмисно.
- **Сортування і ручний DnD у таблиці взаємовиключні.** Поки активне сортування за колонкою, drag вимкнено (інакше наступний ререндер за сортуванням «відкидав» би перетягнутий рядок). Третій клік по заголовку повертає в ручний режим, де DnD знову працює.
- **`order` наскрізний по всьому проекту.** І таблиця (плоский список), і канбан (порядок у межах колонки) читають один спільний `order`, який після будь-якого drop перераховується наскрізно todo → in_progress → done. Це робить синхронізацію між режимами тривіальною — обидва дивляться на той самий масив стору.
